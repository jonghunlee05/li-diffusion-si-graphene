### Header correction (metadata only — no parameter values changed)

As received, the file's section-count headers were carried over from a larger
parent ("Volkswagen") force field and did not match the number of Li/Si/C
entries actually present, so LAMMPS read past the real data and aborted. Every
entry was cross-checked against the **Supporting Information** of the source
paper (Tab. S.2–S.6) and found COMPLETE and value-identical. Only the counts
were corrected:

| section | header (as received) | corrected to | SI table |
|---------|----------------------|--------------|----------|
| atoms | 7 | **3** (Li, Si, C) | S.2 |
| bonds | 20 | **6** (all pairs) | S.3 |
| off-diagonal | 14 | **3** (all cross pairs) | S.4 |
| angles | 62 | **11** | S.5 |
| torsions | 14 | **1** (C–C–C–C) | S.6 |
| h-bonds | 1 (+ dummy `0 0 0` line) | **0** (no H in Li/Si/C) | — |

No `De`, `Θo`, `V1`, etc. parameter value was added, removed, or altered. Spot
checks vs. the SI matched exactly (e.g. Li roσ 2.1097; Si–C bond Deσ 109.7712;
Si–Li off-diag Dij 0.3183; Si–Si–Si angle Θo 74.0968; C–C–C–C V1/V2
2.5000/32.8238). The SI confirms "Force field version from January 12, 2022",
matching the file header.

### Atom order (CONFIRMED from the file)

The ReaxFF file's atom block (the "Nr of atoms" section) lists elements in this
INTERNAL order:

```
elements in ReaxFF file (in file order):  Li  Si  C
   atom 1 = Li   (mass 6.9410)
   atom 2 = Si   (mass 28.0600)
   atom 3 = C    (mass 12.0000)
```

> NOTE: this internal order is **Li, Si, C** — which is the REVERSE of our LAMMPS
> type order. That is fine: LAMMPS `pair_coeff` matches element **names**, not
> the file's internal index, so the file order does not need to match the data
> file. See the mapping below.

> The **order inside the ReaxFF file does not by itself set the LAMMPS mapping.**
> In LAMMPS, `pair_coeff` maps **LAMMPS atom types → element labels**, and the
> labels must each match an element *name present in the file*. What fixes the
> order is your **data file's atom-type numbering**.

### LAMMPS pair_coeff mapping (for the smoke test)

Our smoke-test data file (`structures/reaxff_smoke_test.data`, built by
`structures/build_reaxff_smoke_test.py`) defines:

```
type 1 = C,  type 2 = Si,  type 3 = Li
```

So the LAMMPS input uses:

```
pair_coeff * * force_fields/ffield.reax.LiSiC_OlououGuifo2023.txt C Si Li
```

This is correct **even though** the file's internal order is Li, Si, C: the three
labels `C`, `Si`, `Li` correspond to LAMMPS types 1, 2, 3 (our data file order),
and LAMMPS locates each named element inside the file regardless of its position.

If you change the type order in the builder, change the `pair_coeff` element
order to match. The element labels (`C`, `Si`, `Li`) must each appear in the
ReaxFF file's element list; if the file names them differently, adjust the
labels — **not** the data file.
